# Phase 4 Postgres Schema

This document describes the actual Phase 4 migration contract created by
`mm-chat/backend/migrations/001_initial_schema.*.sql` plus the Phase 4.5
runtime-wiring boundary around Postgres connectivity, readiness, and migration
execution. It is a schema and data boundary reference only; DB repositories and
Compose files are separate later work.

## 1. Scope

Phase 4 makes Postgres the source of truth for structured server data:

```text
users -> sessions
users -> provider_configs
users -> conversations -> messages -> message_attachments -> files
users/sessions/actions -> audit_logs
```

In scope:

- User records and durable session revocation state.
- Provider configuration metadata and encrypted-secret references.
- Conversations, messages, output block JSON, retry idempotency keys, and
  message status.
- File ownership and metadata only; file bytes remain out of Postgres.
- Append-only audit log rows for security and operational events.

Out of scope:

- MinIO/S3 file-byte storage, Redis, RAG indexes, browser import, DB repository
  CRUD code, automatic migrations during API startup, and committed Docker
  Compose assets.

## 2. Baseline Conventions

| Convention | Phase 4 choice |
| --- | --- |
| IDs | `UUID` primary keys generated by the Go application; migrations do not create UUID extensions or call DB UUID generators. |
| Timestamps | `created_at` / `updated_at` use `TIMESTAMPTZ DEFAULT now()` and are maintained by the application after insert. |
| Soft delete | User-facing records use nullable `deleted_at` where lifecycle recovery matters. |
| JSON | `metadata` and `config` use `JSONB` with object checks; message `output_blocks` uses a JSONB array check. |
| Retry safety | `conversations.idempotency_key` and `messages.idempotency_key` are optional unique retry guards scoped to the owner/thread. |
| Secrets | Store token hashes or encrypted secret references only; never plaintext provider keys or bearer tokens. |
| Files | `files` stores metadata and private `object_key` references only, not bytes. |

## 3. Tables

### `users`

Purpose: owner boundary for server-mode data.

Key columns:

- `id UUID PRIMARY KEY`
- `email TEXT UNIQUE` with blank-value guard when present
- `display_name`, `avatar_url`
- `created_at`, `updated_at`, `deleted_at`

Notes:

- Authentication fields such as password hashes or external identities are not
  in the initial migration; add them in a later explicit auth migration.
- Deleting a user cascades owned sessions, conversations, provider configs,
  files, messages through conversations, and attachment links.

### `sessions`

Purpose: canonical session records and revocation state. Redis may later cache
lookups, but Postgres remains authoritative.

Key columns:

- `id UUID PRIMARY KEY`
- `user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE`
- `token_hash TEXT NOT NULL UNIQUE`
- `user_agent`, `ip_address INET`
- `expires_at`, `revoked_at`, `created_at`, `updated_at`

Indexes:

- `idx_sessions_user_id`
- `idx_sessions_expires_at`
- `idx_sessions_active_user` partial index for non-revoked sessions

### `provider_configs`

Purpose: server-side provider configuration metadata. Browser contracts must not
receive provider secrets.

Key columns:

- `id UUID PRIMARY KEY`
- `user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE`
- `provider_id TEXT NOT NULL`
- `label TEXT NOT NULL DEFAULT ''`
- `encrypted_secret_ref TEXT` for encrypted secret storage or secret-manager
  references
- `config JSONB NOT NULL DEFAULT '{}'::jsonb`
- `created_at`, `updated_at`, `deleted_at`

Indexes:

- `idx_provider_configs_user_provider`
- `idx_provider_configs_user_active` partial index for active configs

Boundary:

- `encrypted_secret_ref` is not a plaintext key. Public APIs should expose only
  safe metadata such as provider ID, label, and capability state.

### `conversations`

Purpose: durable chat thread metadata and list surface.

Key columns:

- `id UUID PRIMARY KEY`
- `user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE`
- `title TEXT NOT NULL DEFAULT ''`
- `status TEXT NOT NULL DEFAULT 'active'` with `active|archived|deleted` check
- `model_provider`, `model_id`, `system_prompt`
- `idempotency_key TEXT` with non-blank guard when present
- `metadata JSONB NOT NULL DEFAULT '{}'::jsonb`
- `created_at`, `updated_at`, `deleted_at`

Indexes / constraints:

- `idx_conversations_user_updated`
- `idx_conversations_user_active`
- `idx_conversations_status`
- `idx_conversations_user_idempotency` unique partial index

Boundary:

- Conversation APIs must scope reads and writes by `user_id` before loading
  child messages.
- `metadata` must not store provider API keys or object-store credentials.

### `messages`

Purpose: canonical ordered messages and streaming completion state.

Key columns:

- `id UUID PRIMARY KEY`
- `conversation_id UUID NOT NULL REFERENCES conversations(id) ON DELETE CASCADE`
- `user_id UUID REFERENCES users(id) ON DELETE SET NULL`
- `parent_message_id UUID REFERENCES messages(id) ON DELETE SET NULL`
- `sequence_no INTEGER NOT NULL` with non-negative check and unique
  `(conversation_id, sequence_no)`
- `role TEXT NOT NULL` with `system|user|assistant|tool` check
- `status TEXT NOT NULL DEFAULT 'completed'` with
  `pending|streaming|completed|failed|cancelled` check
- `content TEXT NOT NULL DEFAULT ''`
- `model_provider`, `model_id`, `provider_message_id`
- `idempotency_key TEXT`
- `output_blocks JSONB NOT NULL DEFAULT '[]'::jsonb`
- `error_code`, `error_message`
- `metadata JSONB NOT NULL DEFAULT '{}'::jsonb`
- `created_at`, `updated_at`, `completed_at`, `deleted_at`

Indexes / constraints:

- `messages_conversation_sequence_unique`
- `idx_messages_conversation_created`
- `idx_messages_conversation_role`
- `idx_messages_status`
- `idx_messages_user_id`
- `idx_messages_parent_message_id`
- `idx_messages_conversation_idempotency` unique partial index

Boundary:

- The backend should persist the user message before provider streaming begins.
- Assistant messages can start as `streaming` and finish as `completed`,
  `failed`, or `cancelled`.
- Store stable output fields and scrubbed metadata, not raw provider responses
  or secrets.

### `files`

Purpose: metadata for uploaded/imported files. Bytes live in MinIO/S3/local
object storage later, not in Postgres rows.

Key columns:

- `id UUID PRIMARY KEY`
- `user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE`
- `original_filename TEXT NOT NULL`
- `mime_type TEXT NOT NULL DEFAULT 'application/octet-stream'`
- `byte_size BIGINT NOT NULL` with non-negative check
- `sha256 TEXT NOT NULL` with lowercase 64-char hex check
- `storage_backend TEXT NOT NULL DEFAULT 'minio'` with `local|minio|s3` check
- `object_key TEXT NOT NULL UNIQUE`; it must be server-generated and not start
  with `/`
- `upload_status TEXT NOT NULL DEFAULT 'available'` with
  `pending|available|failed|deleted` check
- `metadata JSONB NOT NULL DEFAULT '{}'::jsonb`
- `created_at`, `updated_at`, `deleted_at`

Indexes:

- `idx_files_user_created`
- `idx_files_sha256`
- `idx_files_upload_status`
- `idx_files_user_active`

Boundary:

- Browser-visible responses return `file_id` or backend download URLs, never
  raw `object_key`, bucket names, or direct object-store endpoints.
- Database cascades only metadata. Object bytes still need application cleanup
  or lifecycle jobs in the MinIO/S3 phase.

### `message_attachments`

Purpose: explicit link between messages and file metadata.

Key columns:

- `id UUID PRIMARY KEY`
- `message_id UUID NOT NULL REFERENCES messages(id) ON DELETE CASCADE`
- `file_id UUID NOT NULL REFERENCES files(id) ON DELETE CASCADE`
- `user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE`
- `display_order INTEGER NOT NULL DEFAULT 0`
- `purpose TEXT NOT NULL DEFAULT 'input'` with
  `input|output|image|knowledge_source` check
- `created_at`, `updated_at`

Indexes / constraints:

- unique `(message_id, file_id)`
- `idx_message_attachments_file_id`
- `idx_message_attachments_user_id`

Boundary:

- Application code must verify message owner and file owner match before insert.
- Deleting a message removes only the attachment link; deleting file metadata is
  a separate product decision unless it cascades through user deletion.

### `audit_logs`

Purpose: append-only security and operations trail.

Key columns:

- `id UUID PRIMARY KEY`
- nullable references: `actor_user_id`, `session_id`, `conversation_id`,
  `message_id`, `file_id`; all use `ON DELETE SET NULL` to preserve evidence
- `actor_type TEXT NOT NULL DEFAULT 'user'` with
  `anonymous|user|admin|system|service` check
- `action TEXT NOT NULL`
- `resource_type TEXT NOT NULL`, `resource_id UUID`
- `request_id TEXT`
- `outcome TEXT NOT NULL DEFAULT 'success'` with
  `success|denied|failed|cancelled` check
- `ip_address INET`, `user_agent`
- `metadata JSONB NOT NULL DEFAULT '{}'::jsonb`
- `created_at`, `updated_at`

Indexes:

- `idx_audit_logs_created_at`
- `idx_audit_logs_actor_created`
- `idx_audit_logs_session_created`
- `idx_audit_logs_resource`
- `idx_audit_logs_outcome_created`
- `idx_audit_logs_request_id`

Boundary:

- Corrective history should be written as new audit events, not by mutating old
  audit rows.
- Do not store API keys, bearer tokens, raw file bytes, or unbounded prompt / raw
  provider payloads in audit `metadata`.

## 4. Phase 4.5 Runtime Wiring Boundary

The app-table schema above stays unchanged in Phase 4.5. Runtime wiring adds the
operational boundary between the Go backend process and Postgres:

- Backend reads `DATABASE_URL`, `DB_MAX_OPEN_CONNS`, `DB_MAX_IDLE_CONNS`, and
  `DB_CONN_MAX_LIFETIME`.
- `DATABASE_URL` empty means DB disabled mode: the API can start without
  Postgres and `/ready` remains `200 OK` for the skeleton/runtime path.
- `DATABASE_URL` non-empty means DB enabled mode: startup creates a Postgres
  connector with `github.com/jackc/pgx/v5` through `database/sql`, runs
  `PingContext`, and fails fast before serving HTTP if the initial ping fails.
  `/ready` also pings DB and returns `503 Service Unavailable` on later DB ping
  failure.
- `/health` remains process liveness. `/ready` is the dependency gate once DB is
  enabled.
- API startup must not run migrations automatically. Schema changes are an
  explicit operator/deployment step.
- Migrations are exposed through a Go CLI with embedded SQL. Expected command
  shape is `cd mm-chat/backend && go run ./cmd/migrate up`.
- The runner records applied versions in
  `schema_migrations(version, name, applied_at)`. This table is
  migration-runner metadata, not an app/domain table.

## 5. Migration and Rollback Boundary

- Apply `*.up.sql` in ascending order and `*.down.sql` in descending order.
- `001_initial_schema.down.sql` drops indexes, child tables, then foundational
  tables and leaves zero app tables in a clean development DB.
- No enum types, custom functions, triggers, extensions, or explicit SQL
  transaction control statements are created by the initial migration; the Go
  runner owns transaction boundaries.
- Phase 4.5 migration execution is owned by the Go migration CLI, not API
  startup. Expected source-run command shape is
  `cd mm-chat/backend && go run ./cmd/migrate up`.
- Down migrations are for deliberate rollback/development resets; the draft
  destructive command shape is `go run ./cmd/migrate down --all`.
- The runner records applied versions in
  `schema_migrations(version, name, applied_at)`. This table is
  migration-runner metadata, not an application table in the domain model above.

## 6. Acceptance Checks

Phase 4/4.5 persistence is not complete until implementation work can verify:

- The initial migration applies on an empty Postgres 16 database.
- The down migration removes all Phase 4 app tables in development.
- Constraints reject invalid message roles/statuses, invalid file hashes,
  negative byte sizes, non-object metadata, and non-array output blocks.
- File metadata rows do not require MinIO to be running.
- Frontend rollback to `NEXT_PUBLIC_API_MODE=local` does not delete Postgres
  data.
- With `DATABASE_URL` empty, DB disabled mode starts without Postgres and
  `/ready` remains `200 OK`.
- With `DATABASE_URL` set, startup attempts a Postgres connection and
  `PingContext`, and `/ready` returns `503 Service Unavailable` when DB ping
  fails.
- API startup does not apply migrations automatically; operators can run the
  migration CLI before backend deploy/restart.
