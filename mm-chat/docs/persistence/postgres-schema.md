# Postgres Schema Through Migration 009

This document describes the current schema created by the ordered migrations in
`mm-chat/backend/migrations`, from `001_initial_schema` through
`009_phase15_consent_expiry_materialization`. Phase labels are retained where
they explain rollout history; they do not narrow the current schema or its
implemented repository consumers.

## 1. Scope

Postgres is the source of truth for structured server data:

```text
users -> sessions
users -> provider_configs
users -> conversations -> messages -> message_attachments -> files
users -> import_batches
users/sessions/actions -> audit_logs
users -> credentials/recovery tokens
users -> teams -> memberships/invites -> encrypted mail outbox
users/teams -> knowledge collections -> documents -> versions -> processing jobs
governance heads/profiles -> processing consents -> knowledge jobs/outbox
```

In scope:

- User records and durable session revocation state.
- Provider configuration metadata and encrypted-secret references.
- Conversations, messages, output block JSON, retry idempotency keys, and
  message status.
- Browser import batch idempotency, response replay, and rollback state.
- File ownership and metadata only; file bytes remain out of Postgres.
- Append-only audit log rows for security and operational events.
- Independent credentials and recovery state, Teams, Memberships, Invites, and
  encrypted durable Invite delivery.
- Knowledge Collection/Document/Version lifecycle, ACL and processing revision
  fences, durable Processing Jobs, and transactional Outbox events.
- Immutable Processor Governance Profiles, authoritative Governance Heads,
  Processing Consent history, and one-time expiry materialization.

Out of scope:

- MinIO/S3 file-byte storage, Redis data structures, search/RAG indexes and
  execution, automatic migrations during API startup, and deployment topology.

## 2. Baseline Conventions

| Convention   | Current choice                                                                                                                                                                                      |
| ------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| IDs          | `UUID` primary keys generated by the Go application; migrations do not create UUID extensions or call DB UUID generators.                                                                           |
| Timestamps   | `created_at` / `updated_at` use `TIMESTAMPTZ DEFAULT now()` and are maintained by the application after insert.                                                                                     |
| Soft delete  | User-facing records use nullable `deleted_at` where lifecycle recovery matters.                                                                                                                     |
| JSON         | `metadata` and `config` use `JSONB` with object checks; message `output_blocks` uses a JSONB array check.                                                                                           |
| Retry safety | `conversations.idempotency_key` and `messages.idempotency_key` are optional unique retry guards scoped to the owner/thread; `import_batches.idempotency_key` replays exact browser import packages. |
| Secrets      | Store token hashes or encrypted secret references only; never plaintext provider keys or bearer tokens.                                                                                             |
| Files        | `files` stores metadata and private `object_key` references only, not bytes.                                                                                                                        |

## 3. Migration History

| Migration                                    | Current schema effect                                                                                                                                                                                                                                      |
| -------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `001_initial_schema`                         | Creates users, sessions, provider configs, conversations, messages, files, message attachments, and audit logs.                                                                                                                                            |
| `002_messages_run_id_index`                  | Adds the assistant `metadata.runId` cancellation lookup index.                                                                                                                                                                                             |
| `003_import_batches`                         | Adds durable browser-import idempotency, replay, and rollback state.                                                                                                                                                                                       |
| `004_phase15_identity_knowledge_acl`         | Adds account status, credentials and recovery tokens; Teams, Memberships, and Invites; Knowledge Collections, Documents, and Versions; query-consent revision state; Governance Profiles and Heads; Processing Consent history; and the Knowledge Outbox.  |
| `005_phase15_team_services`                  | Adds scoped Team/Invite idempotency, one encrypted durable mail-delivery row per Invite, pending-Invite uniqueness, and `ON DELETE RESTRICT` Membership-to-User fencing.                                                                                   |
| `006_phase15_knowledge_services`             | Adds Collection presentation/replay metadata, Document/Version visibility and replay metadata, one-nonterminal-Version enforcement, Consent lookup/binding support, and durable stage-specific Processing Jobs with authority snapshots and purge fencing. |
| `007_phase15_knowledge_deletion`             | Uses `CREATE UNIQUE INDEX IF NOT EXISTS` to reconcile databases created from a short-lived `006` variant without the purge fence; current `006` already creates the same index.                                                                            |
| `008_phase15_governance_immutability`        | Adds a trigger that rejects every `UPDATE` or `DELETE` of a Processor Governance Profile.                                                                                                                                                                  |
| `009_phase15_consent_expiry_materialization` | Adds `expiry_materialized_at` and an index for current granted Consent rows whose finite expiry is due and not yet materialized.                                                                                                                           |

Published migration pairs are immutable and applied in numeric order. Migration
SQL contains no transaction-control statements; the Go runner wraps each schema
change and its metadata update in one transaction.

## 4. Tables

### `users`

Purpose: owner boundary for server-mode data.

Key columns:

- `id UUID PRIMARY KEY`
- `email TEXT UNIQUE` with blank-value guard when present
- `display_name`, `avatar_url`
- `created_at`, `updated_at`, `deleted_at`

Notes:

- Password hashes are intentionally absent from `users`; migration `004` stores
  them in the one-to-one `user_credentials` table.
- Deleting a user cascades owned sessions, conversations, provider configs,
  files, messages through conversations, and attachment links.

### `sessions`

Purpose: canonical session records and revocation state. Redis may cache
browser-safe lookup snapshots and short-lived revocation hints, but Postgres
remains authoritative.

Key columns:

- `id UUID PRIMARY KEY`
- `user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE`
- `token_hash TEXT NOT NULL UNIQUE`
- `user_agent`, `ip_address INET`
- `expires_at`, `revoked_at`, `created_at`, `updated_at`

Indexes:

- `idx_sessions_user_id`
- `idx_sessions_expires_at`
- `idx_sessions_active_user` partial index for non-revoked sessions

### `provider_configs`

Purpose: server-side provider configuration metadata. Browser contracts must not
receive provider secrets.

Key columns:

- `id UUID PRIMARY KEY`
- `user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE`
- `provider_id TEXT NOT NULL`
- `label TEXT NOT NULL DEFAULT ''`
- `encrypted_secret_ref TEXT` for encrypted secret storage or secret-manager
  references
- `config JSONB NOT NULL DEFAULT '{}'::jsonb`
- `created_at`, `updated_at`, `deleted_at`

Indexes:

- `idx_provider_configs_user_provider`
- `idx_provider_configs_user_active` partial index for active configs

Boundary:

- `encrypted_secret_ref` is not a plaintext key. Public APIs should expose only
  safe metadata such as provider ID, label, and capability state.

### `conversations`

Purpose: durable chat thread metadata and list surface.

Key columns:

- `id UUID PRIMARY KEY`
- `user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE`
- `title TEXT NOT NULL DEFAULT ''`
- `status TEXT NOT NULL DEFAULT 'active'` with `active|archived|deleted` check
- `model_provider`, `model_id`, `system_prompt`
- `idempotency_key TEXT` with non-blank guard when present
- `metadata JSONB NOT NULL DEFAULT '{}'::jsonb`
- `created_at`, `updated_at`, `deleted_at`

Indexes / constraints:

- `idx_conversations_user_updated`
- `idx_conversations_user_active`
- `idx_conversations_status`
- `idx_conversations_user_idempotency` unique partial index

Boundary:

- Conversation APIs must scope reads and writes by `user_id` before loading
  child messages.
- `metadata` must not store provider API keys or object-store credentials.

### `messages`

Purpose: canonical ordered messages and streaming completion state.

Key columns:

- `id UUID PRIMARY KEY`
- `conversation_id UUID NOT NULL REFERENCES conversations(id) ON DELETE CASCADE`
- `user_id UUID REFERENCES users(id) ON DELETE SET NULL`
- `parent_message_id UUID REFERENCES messages(id) ON DELETE SET NULL`
- `sequence_no INTEGER NOT NULL` with non-negative check and unique
  `(conversation_id, sequence_no)`
- `role TEXT NOT NULL` with `system|user|assistant|tool` check
- `status TEXT NOT NULL DEFAULT 'completed'` with
  `pending|streaming|completed|failed|cancelled` check
- `content TEXT NOT NULL DEFAULT ''`
- `model_provider`, `model_id`, `provider_message_id`
- `idempotency_key TEXT`
- `output_blocks JSONB NOT NULL DEFAULT '[]'::jsonb`
- `error_code`, `error_message`
- `metadata JSONB NOT NULL DEFAULT '{}'::jsonb`
- `created_at`, `updated_at`, `completed_at`, `deleted_at`

Indexes / constraints:

- `messages_conversation_sequence_unique`
- `idx_messages_conversation_created`
- `idx_messages_conversation_role`
- `idx_messages_status`
- `idx_messages_user_id`
- `idx_messages_parent_message_id`
- `idx_messages_conversation_idempotency` unique partial index
- `idx_messages_assistant_run_id` expression partial index on
  `(metadata ->> 'runId')` for assistant-run cancellation lookup

Boundary:

- The backend should persist the user message before provider streaming begins.
- Assistant messages can start as `streaming` and finish as `completed`,
  `failed`, or `cancelled`.
- Streaming run identity is stored as `metadata.runId` on assistant messages
  until a durable run table is introduced.
- Store stable output fields and scrubbed metadata, not raw provider responses
  or secrets.

### `files`

Purpose: metadata for uploaded/imported files. Bytes live in private object
storage, not in Postgres rows.

Key columns:

- `id UUID PRIMARY KEY`
- `user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE`
- `original_filename TEXT NOT NULL`
- `mime_type TEXT NOT NULL DEFAULT 'application/octet-stream'`
- `byte_size BIGINT NOT NULL` with non-negative check
- `sha256 TEXT NOT NULL` with lowercase 64-char hex check
- `storage_backend TEXT NOT NULL DEFAULT 'minio'` with `local|minio|s3` check
- `object_key TEXT NOT NULL UNIQUE`; it must be server-generated and not start
  with `/`
- `upload_status TEXT NOT NULL DEFAULT 'available'` with
  `pending|available|failed|deleted` check
- `metadata JSONB NOT NULL DEFAULT '{}'::jsonb`
- `created_at`, `updated_at`, `deleted_at`

Indexes:

- `idx_files_user_created`
- `idx_files_sha256`
- `idx_files_upload_status`
- `idx_files_user_active`

Boundary:

- Browser-visible responses return `file_id` or backend download URLs, never
  raw `object_key`, bucket names, or direct object-store endpoints.
- Database cascades only metadata. Object bytes still need application cleanup
  or lifecycle jobs in the MinIO/S3 phase.

### `message_attachments`

Purpose: explicit link between messages and file metadata.

Key columns:

- `id UUID PRIMARY KEY`
- `message_id UUID NOT NULL REFERENCES messages(id) ON DELETE CASCADE`
- `file_id UUID NOT NULL REFERENCES files(id) ON DELETE CASCADE`
- `user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE`
- `display_order INTEGER NOT NULL DEFAULT 0`
- `purpose TEXT NOT NULL DEFAULT 'input'` with
  `input|output|image|knowledge_source` check
- `created_at`, `updated_at`

Indexes / constraints:

- unique `(message_id, file_id)`
- `idx_message_attachments_file_id`
- `idx_message_attachments_user_id`

Boundary:

- Application code must verify message owner and file owner match before insert.
- Deleting a message removes only the attachment link; deleting file metadata is
  a separate product decision unless it cascades through user deletion.

### `import_batches`

Purpose: Phase 8 browser import idempotency, replay, and rollback metadata.

Key columns:

- `id UUID PRIMARY KEY`
- `user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE`
- `idempotency_key TEXT NOT NULL`
- `package_hash` and `manifest_hash` as lowercase SHA-256 hex strings
- `status TEXT NOT NULL DEFAULT 'pending'` with
  `pending|completed|rolled_back` check
- `response JSONB NOT NULL DEFAULT '{}'::jsonb`
- `created_at`, `updated_at`, `completed_at`, `rolled_back_at`

Indexes / constraints:

- `idx_import_batches_user_idempotency` unique `(user_id, idempotency_key)`
- `idx_import_batches_user_created`
- `idx_import_batches_status`

Boundary:

- Same `idempotency_key` plus same package/manifest hash returns the stored
  completed response instead of creating duplicate conversations/messages.
- Same `idempotency_key` with different package bytes returns
  `409 IDEMPOTENCY_CONFLICT`.
- Imported conversations, messages, and files carry `metadata.import.batchId`
  plus original client IDs so `DELETE /v1/import/browser/{batchId}` can locate
  and roll back the batch.
- Browser import writes ZIP-backed file bytes through private object storage,
  inserts `files` metadata rows, and links file-backed message attachments via
  `message_attachments`; remote attachments remain metadata-only.

### `audit_logs`

Purpose: append-only security and operations trail.

Key columns:

- `id UUID PRIMARY KEY`
- nullable references: `actor_user_id`, `session_id`, `conversation_id`,
  `message_id`, `file_id`; all use `ON DELETE SET NULL` to preserve evidence
- `actor_type TEXT NOT NULL DEFAULT 'user'` with
  `anonymous|user|admin|system|service` check
- `action TEXT NOT NULL`
- `resource_type TEXT NOT NULL`, `resource_id UUID`
- `request_id TEXT`
- `outcome TEXT NOT NULL DEFAULT 'success'` with
  `success|denied|failed|cancelled` check
- `ip_address INET`, `user_agent`
- `metadata JSONB NOT NULL DEFAULT '{}'::jsonb`
- `created_at`, `updated_at`

Indexes:

- `idx_audit_logs_created_at`
- `idx_audit_logs_actor_created`
- `idx_audit_logs_session_created`
- `idx_audit_logs_resource`
- `idx_audit_logs_outcome_created`
- `idx_audit_logs_request_id`

Boundary:

- Corrective history should be written as new audit events, not by mutating old
  audit rows.
- Do not store API keys, bearer tokens, raw file bytes, or unbounded prompt / raw
  provider payloads in audit `metadata`.

### Identity, Team, and Invite delivery tables

Migration `004` adds `users.account_status`, one-to-one `user_credentials`,
single-use `credential_recovery_tokens`, `teams`, `team_memberships`, and
`team_invites`. Credentials contain Argon2id hashes; invitation and recovery
records store token hashes rather than raw tokens.

Migration `005` adds bounded, scoped idempotency keys to Team and Invite
creation and creates `identity_mail_outbox`. Each Invite has at most one mail
row; mailbox, raw invitation token, and acceptance URL live only in
authenticated ciphertext with key/nonce uniqueness. Delivery lifecycle is
`pending|processing|sent|failed|cancelled`, with lease, retry, attempt, and
terminal-state constraints. The Membership-to-User foreign key uses
`ON DELETE RESTRICT` so account deletion cannot silently erase Team authority.

### Knowledge ACL entities

- `knowledge_collections` records immutable Personal-owner or Team scope plus
  `acl_revision`, `visibility_epoch`, and `collection_processing_revision`.
  Migration `006` adds display fields and creator-scoped idempotent replay
  metadata.
- `knowledge_documents` is the stable logical identity. Its authoritative
  lifecycle is `processing|active|tombstoned|deleted`; an Active row points to
  an Active Version in the same Document through deferred composite foreign
  keys. Migration `006` adds Document visibility and bind replay metadata.
- `knowledge_document_versions` binds an immutable File/source version/hash.
  Its lifecycle is
  `uploaded|processing|failed|active|tombstoned|purging|deleted`. Migration
  `006` adds creator-scoped replay metadata, sanitized `error_code`, and a
  partial unique index allowing only one `uploaded|processing` Version per
  Document.

### Governance and Consent

- `processor_governance_profiles` stores versioned Processor endpoint/model,
  purposes, data types, region, retention, deletion, training-use, and manifest
  declarations. Migration `008` makes every row immutable: a trigger rejects
  both `UPDATE` and `DELETE`; policy changes require a new Profile.
- `processor_governance_heads` is the mutable authoritative pointer to one
  Approved Profile/Revision or to a Disabled state, with its own monotonic head
  revision.
- `processing_consents` stores append-style Collection or User Query decisions
  pinned to Governance Profile, Governance Revision, and Head Revision. Current
  decisions are selected by `superseded_at IS NULL` and constrained to one
  current row per subject/Processor.
- Migration `009` adds nullable `expiry_materialized_at`. It may be set only on
  a granted row with finite expiry and only at or after `expires_at`.
  `idx_processing_consents_expiry_due` selects current, granted, expired rows
  whose expiry side effects have not yet been materialized. This marker records
  one-time processing; it does not rewrite the original Consent decision.
- `user_query_consent_state.query_consent_revision` and
  `knowledge_collections.collection_processing_revision` provide independent
  invalidation fences for query and Collection processing authority.

### `knowledge_processing_jobs`

Migration `006` creates durable stage-specific work with:

- stages `parse|passage_embedding|purge`, operations
  `initial|replace|reprocess|purge`, and lifecycle
  `pending|processing|succeeded|failed|cancelled`;
- immutable bindings to Collection, Document, Version, and File, plus optional
  predecessor Job linkage;
- pinned Governance Profile/Head and Collection Consent for external processing
  stages; purge Jobs must have no external processor or Consent authority;
- captured Collection ACL/visibility/processing revisions and Document
  visibility epoch, so workers can reject stale work;
- bounded idempotency scope/key plus canonical request hash, lease/attempt
  fields, and strict terminal-state constraints.

The partial unique index `idx_knowledge_processing_jobs_purge_fence` permits at
most one purge Job for each
`(document_id, document_version_id, document_visibility_epoch)` fence where
`stage = 'purge' AND operation = 'purge'`. Current migration `006` creates the
index; migration `007` idempotently repairs only databases that passed through
the short-lived `006` variant without it.

### `knowledge_outbox`

The Knowledge Outbox is transactional durable work/event state keyed by an
allocation-order `BIGSERIAL` cursor and unique event identity. Consumers must
rescan claimable rows and treat the cursor as allocation order, not transaction
commit order; only a contiguous applied prefix is a safe high-watermark.

## 5. Historical Repository Activation Boundary

Phase 5.1 first activated repositories for `users`, `conversations`, and
`messages`; Phase 5.2 added assistant streaming rows, Phase 5.4 run
cancellation, and Phase 6.3 file attachments. Those labels explain rollout
order only. The current runtime also consumes auth/session, Team, Knowledge,
Governance, Consent, Job, and Outbox tables introduced through migration `009`.
DB-backed endpoints fail with `503 DATABASE_REQUIRED` when DB wiring is
disabled; request handling never applies migrations.

## 6. Runtime Wiring and Migration Runner

The historical Phase 4.5 work established the operational boundary between the
Go backend process and Postgres:

- Backend reads `DATABASE_URL`, `DB_MAX_OPEN_CONNS`, `DB_MAX_IDLE_CONNS`, and
  `DB_CONN_MAX_LIFETIME`.
- `DATABASE_URL` empty means DB disabled mode: the API can start without
  Postgres and `/ready` remains `200 OK` for the skeleton/runtime path.
- `DATABASE_URL` non-empty means DB enabled mode: startup creates a Postgres
  connector with `github.com/jackc/pgx/v5` through `database/sql`, runs
  `PingContext`, and fails fast before serving HTTP if the initial ping fails.
  `/ready` also pings DB and returns `503 Service Unavailable` on a subsequent
  DB ping failure.
- `/health` remains process liveness. `/ready` is the dependency gate once DB is
  enabled.
- API startup must not run migrations automatically. Schema changes are an
  explicit operator/deployment step.
- Migrations are exposed through a Go CLI with embedded SQL. Expected command
  shape is `cd mm-chat/backend && go run ./cmd/migrate up`.
- The runner records applied versions in
  `schema_migrations(version, name, checksum, applied_at)`. The SHA-256 checksum
  covers migration identity and both SQL directions. `up`, `down`, and explicit
  legacy `baseline` operations hold one PostgreSQL advisory lock across metadata
  setup, name/checksum validation, and the complete requested operation. This
  table is migration-runner metadata, not an app/domain table.

## 7. Migration and Rollback Boundary

- Apply `*.up.sql` in ascending order and `*.down.sql` in descending order.
- `001_initial_schema.down.sql` drops indexes, child tables, then foundational
  tables and leaves zero app tables in a clean development DB.
- No enum types, custom functions, triggers, extensions, or explicit SQL
  transaction control statements are created by the initial migration; the Go
  runner owns transaction boundaries.
- Migration execution is owned by the Go migration CLI, not API startup.
  Expected source-run command shape is
  `cd mm-chat/backend && go run ./cmd/migrate up`.
- Down migrations are for deliberate rollback/development resets; the
  destructive full-reset command is `go run ./cmd/migrate down --all`.
- The runner records applied versions in
  `schema_migrations(version, name, checksum, applied_at)`. Rows created before
  checksum tracking require an explicit `go run ./cmd/migrate baseline` after
  the operator verifies the embedded migration source; routine deployment must
  not automate baseline. Name/checksum drift fails closed. This table is
  migration-runner metadata, not an application table in the domain model
  above.

## 8. Acceptance Checks

Current persistence verification covers:

- Migrations `001` through `009` apply in order on an empty Postgres 16 database,
  and `down --all` reverses app-schema objects in development while retaining
  migration-runner metadata.
- Constraints reject invalid message roles/statuses, invalid file hashes,
  negative byte sizes, non-object metadata, and non-array output blocks.
- Knowledge tests enforce one nonterminal Version, pinned Processing Job
  authority, and one purge Job per
  `(document_id, document_version_id, document_visibility_epoch)` fence.
- Governance Profile `UPDATE`/`DELETE` fails after migration `008`, and Consent
  expiry work is selected and marked once through migration `009`.
- Applied migration name/checksum drift fails closed; explicit legacy baseline
  fills only reviewed missing checksums; concurrent migration operations
  serialize on the advisory lock.
- File metadata rows do not require MinIO to be running.
- With `DATABASE_URL` empty, DB disabled mode starts without Postgres and
  `/ready` remains `200 OK`.
- With `DATABASE_URL` set, startup attempts a Postgres connection and
  `PingContext`, and `/ready` returns `503 Service Unavailable` when DB ping
  fails.
- API startup does not apply migrations automatically; operators can run the
  migration CLI before backend deploy/restart.
