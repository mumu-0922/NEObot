# mm-chat backend migrations

This directory contains ordered SQL migrations for the `mm-chat` Go backend
Postgres persistence layer.

## Naming

Use paired files with a zero-padded sequence number and a short slug:

```text
NNN_short_description.up.sql
NNN_short_description.down.sql
```

Rules:

- `NNN` is monotonic and never reused.
- Each pair represents one reversible schema change.
- Keep SQL Postgres-compatible and avoid third-party extensions.
- UUID values are generated by the Go application before insert. Migrations may
  use the Postgres `uuid` type, but must not call database-side UUID generator
  functions or create extension dependencies.

## Apply and rollback order

Apply `*.up.sql` files in ascending sequence order:

```text
001_initial_schema.up.sql
002_next_change.up.sql
...
```

Rollback with matching `*.down.sql` files in descending sequence order:

```text
...
002_next_change.down.sql
001_initial_schema.down.sql
```

Down migrations must remove dependent objects first: application-created indexes,
child tables, custom types if any, then foundational tables. The initial schema
uses `CHECK` constraints instead of custom enum types, so there are no custom
Postgres types to clean up.

## Verification

Preferred local validation uses the official Postgres image and `psql` with
`ON_ERROR_STOP=1`:

```bash
docker run --rm -d \
  --name mm-chat-pg-migration-test \
  -e POSTGRES_PASSWORD=postgres \
  -e POSTGRES_DB=mm_chat \
  postgres:16-alpine

until docker exec mm-chat-pg-migration-test \
  pg_isready -U postgres -d mm_chat >/dev/null 2>&1; do
  sleep 1
done

cat mm-chat/backend/migrations/001_initial_schema.up.sql | \
  docker exec -i mm-chat-pg-migration-test \
    psql -U postgres -d mm_chat -v ON_ERROR_STOP=1

cat mm-chat/backend/migrations/001_initial_schema.down.sql | \
  docker exec -i mm-chat-pg-migration-test \
    psql -U postgres -d mm_chat -v ON_ERROR_STOP=1

docker rm -f mm-chat-pg-migration-test
```

If Docker is unavailable, at minimum run the same `psql` commands against any
Postgres 16 database that can be destroyed after the check.

## Storage boundaries

Postgres is the source of truth for structured records:

- users and canonical session records
- provider configuration metadata and encrypted-secret references
- conversations and messages
- file ownership and metadata
- append-only audit logs

Redis is only for short-lived state such as cache entries, rate limits, stream
cancellation flags, and temporary jobs. Redis must not be the only copy of
canonical users, conversations, messages, files, or audit records.

MinIO/S3-compatible storage holds file bytes. Postgres stores only metadata such
as owner, MIME type, size, hash, backend, and internal object key. Raw filenames
are display metadata only and must never be used as object keys. Browser clients
must not receive bucket names, object keys, or direct object-store URLs unless a
future contract explicitly changes that boundary.
